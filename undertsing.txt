An authentication expert (Speaker D) explains the exact workflow required to secure the API:
1. **Frontend Authentication:** The frontend application currently handles user logins via Microsoft OAuth, which automatically generates an encoded **Bearer token** upon successful validation. 
2. **Middleware Interception:** The frontend team needs to pass this Bearer token inside the Authorization header of their API requests. On the backend, Speaker A must create a routing middleware that intercepts every incoming API request before it reaches the endpoint.
3. **Microsoft Graph API Validation:** Once the backend middleware extracts the Bearer token, it will forward the token to the **Microsoft Graph API**. The Graph API will validate the token and return the authenticated user's details, such as their name, employee ID, and email ID.
4. **Database Verification:** As a final step, the backend will verify the returned email ID against its internal database. If the user is recognized, the API will serve the requested data. If the token is missing, invalid, or the user is not in the database, the middleware will automatically block the request and return an unauthorized error. 

By implementing this, the team ensures that unauthorized or direct browser hits to the API will immediately be rejected, satisfying the "deny by default" requirement.
